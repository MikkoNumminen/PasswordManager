# ADR 0007: Access layer and credentials

## Status

Accepted (Phase 4).

## Context

The sync server needs something that gates blob access. The invariant from
the spec: authentication is never an input to key derivation, and losing an
auth account must not expose vault contents. Two access paths exist with
different exposure: the tailnet (private, primary) and an optional public
web page.

## Decision

Two bearer credentials, both authorizing ciphertext access only:

1. API token. 32 random bytes, hex encoded, generated by
   `password-manager-server token`. The server stores only its SHA-256 hash and
   compares with a constant time comparison (`subtle`). Used by the CLI
   over the tailnet. The client stores the token in its local database;
   it can fetch ciphertext and nothing else.
2. Google ID token (optional, for the public web page). Verified server
   side: RS256 signature against Google's published JWKS, issuer,
   audience (the deployment's OAuth client id), expiry, verified email on
   a configured allowlist.

Neither credential touches key derivation. The vault key comes from the
master password alone. An attacker holding both credentials gets exactly
what a server breach gets: ciphertext and metadata.

The server binds to 127.0.0.1 by default. Tailnet use binds to the
tailscale interface address. Public exposure happens only through an
explicit tunnel (Cloudflare Tunnel or equivalent) that the operator sets
up deliberately.

## Consequences

- Losing the Google account or leaking the API token exposes ciphertext,
  not secrets. Rotating is one `password-manager-server token` run away.
- The public path depends on Google's OIDC infrastructure being reachable
  from the server for JWKS fetches. The tailnet path has no external
  dependency.
- JWT verification uses the `jsonwebtoken` crate. This is the one
  cryptographic dependency outside RustCrypto, it runs only on the server,
  and it verifies third party identity tokens; it never touches vault
  crypto.
