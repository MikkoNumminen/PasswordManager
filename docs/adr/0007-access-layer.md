# ADR 0007: Access layer and credentials

## Status

Accepted (Phase 4). Revised in the access-layer update: identity on the
public path moved from in-app Google OIDC verification to Cloudflare
Access at the edge, and the OIDC code was removed from the server.

## Context

The sync server needs something that gates who can reach it. The invariant
from the spec: authentication is never an input to key derivation, the
server never holds, receives, or releases the vault key, and losing an
auth account must not expose vault contents. Two access paths exist with
different exposure: the tailnet (private, primary) and an optional public
path. Hand-building OAuth token verification inside the server was working
but wrong-shaped: it put identity logic, a JWKS fetch path, and third
party dependencies inside the one process that must stay minimal, and it
ran identity checks after requests already reached the app.

## Decision

Identity is enforced outside the application. The app itself knows exactly
one credential.

1. API token (the only credential the app checks). 32 random bytes, hex
   encoded, generated by `password-manager-server token`. The server
   stores only its SHA-256 hash and compares in constant time (`subtle`).
   Used by every client on every path; it authorizes ciphertext access
   and nothing else.
2. Tailnet (default private path). The server binds 127.0.0.1 or a
   tailnet address; membership in the tailnet is the perimeter. No public
   address exists.
3. Cloudflare Tunnel plus Cloudflare Access (opt-in public path). The
   origin gets no public IP: `cloudflared` makes an outbound connection
   and the hostname (a subdomain of the operator's own domain, kept
   configurable in the tunnel config) routes through Cloudflare. Access
   sits in front with Google as the identity provider and an allowlist
   policy restricted to the operator's email. Identity is checked at the
   edge before any request reaches the app. The Google client id and
   secret live in Cloudflare's dashboard; the tunnel token lives in an
   environment variable or the cloudflared service store. None of it
   exists in this repository or this codebase.

The hard invariant, restated: Cloudflare Access and Google authorize who
can reach the service, and nothing more. They are never inputs to key
derivation. The vault key is derived from the master password client side
(natively or in the wasm client), and the server has no code path that
derives a key or decrypts a record. A valid Google session plus a
reachable server plus a valid API token still yields only ciphertext.

The web client's static assets (page, JavaScript, wasm) are served by the
same axum process from the same machine. No third party hosts the crypto
code the browser executes, so no CDN or static host can serve a modified
client that leaks the master password. The tunnel forwards to the same
origin; Cloudflare proxies bytes but the assets originate here.

## Consequences

- The server crate contains no identity code: no OAuth, no OIDC, no JWKS
  fetching, no Google types, and two fewer dependencies (jsonwebtoken,
  reqwest). Its auth surface is one constant-time token comparison.
- Compromising the Google account or the Cloudflare account exposes
  reachability, not contents: an attacker still faces the API token, and
  with it they get exactly what a server breach gets, ciphertext.
- Rotating the API token is one `password-manager-server token` run;
  rotating edge identity is a Cloudflare policy change, no deploy.
- The public path depends on Cloudflare availability and on trusting
  Cloudflare to enforce the Access policy. The tailnet path has no such
  dependency and stays the default.
- The public hostname is a visible, scannable target even though every
  request must pass Access first. The threat model in the README names
  this.
